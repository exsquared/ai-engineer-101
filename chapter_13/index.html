    Let Your Model Think and Act

    Agents, Tool Use, and the Reasoning Loop

    üß≠ Why Are We Here?

    You‚Äôve built models. You‚Äôve served them with FastAPI. You‚Äôve retrieved documents with RAG.

    But real-world questions don‚Äôt always have obvious answers or clean one-shot responses. Users ask things that require multi-step thinking, calling tools, searching, calculating, retrying.

    This is where agents shine ‚Äî giving LLMs the ability to think step-by-step, take actions, observe outcomes, and adapt.

    We‚Äôre now entering the era of agentic systems.

    üîÅ Before Agents: The Intent Routing Phase

    Even after LLMs arrived, we still built logic-heavy, brittle systems.

    The pattern looked like this:

    1. Ask the LLM to return an intent:
    {"intent": "get_weather", "location": "London"}
    1. Manually route in code:
    if intent == "get_weather":
    return get_weather(location)
elif intent == "convert_currency":
    return convert_currency(from_curr, to_curr)
else:
    return "Sorry, I can't help with that."

    ‚úÖ This worked.
    ‚ùå But it was rigid: - Every new tool = more branching logic
    - You had to trust the intent parser completely
    - Multi-intent or follow-up tasks? Forget it
    - Hard to retry, inspect, or trace reasoning

    The LLM was just an intent extractor ‚Äî all logic lived in your backend.

    This was better than rules, but still far from agentic thinking.

    ‚ö° ReAct + Function Calling Changed Everything

    ReAct (Reason + Act + Observe) lets the model:

    • Think out loud
    • Decide which tool to call
    • Use the result to continue reasoning

    Function calling gives the model access to real tools in a safe, structured way.

    Instead of hardcoded logic, the model says:

    Thought: I need to convert USD to INR.
Action: get_exchange_rate("USD")
Observation: 1 USD = 83 INR
Answer: You‚Äôll get 83 rupees per dollar.

    Now LLMs can orchestrate logic ‚Äî not just output text.

    ü§ñ What Is an Agent?

    An agent is an LLM system that:

    1. Thinks about what it needs to do
    2. Chooses a tool to take action
    3. Observes the result
    4. Repeats or finishes

    This is called an agent loop.

    üß† Agentic Thinking

    Traditional APIs Agentic Systems
    Developer wires logic Model decides flow
    Tools are hardwired Tools are exposed and selected dynamically
    Input ‚Üí logic ‚Üí response Input ‚Üí thoughts ‚Üí actions ‚Üí response

    Agents blur the line between logic and language ‚Äî and put decision-making inside the model.

    üß± Agent Design Patterns

    üß† ReAct Loop

    Thought ‚Üí Action ‚Üí Observation ‚Üí Thought ‚Üí Answer

    üß† Planner‚ÄìExecutor

    • Agent 1: plans steps
    • Agent 2: executes tools

    üß† Agent-as-API-Router

    Let a single LLM route requests to the right API or logic layer based on intent.

    üß† Agent-as-Backend-Brain

    Instead of chaining APIs in microservices, the agent decides what to call, when, and in what order.

    üîß What Are Tools?

    In LangChain or OpenAI:

    A tool is a function the model can call.

    You define:

    @tool
def get_weather(city: str) -> str:
    return "Sunny and 23¬∞C in Delhi"

    And the model decides when to call it ‚Äî with what args.

    üõ†Ô∏è Build a Tool-Using Agent

    from langchain.agents import tool

@tool

    return "1 USD = 83 INR"

    Then:

    tools=[get_exchange_rate],
    llm=llm,
    agent=AgentType.OPENAI_FUNCTIONS,
    verbose=True,

    ‚úÖ Now the model thinks + acts ‚Äî not just responds.

    üß† When Do Agents Replace Microservices?

    Sometimes the LLM + tool layer is your backend logic.

    Task Traditional Microservice Agentic Alternative
    Intent classification Flask route or model LLM with ReAct routing
    Email triage Rules or model ‚Üí API call Agent chooses tool based on message
    Q&A + math + lookup Separate endpoints One agent choosing which to do

    You‚Äôre no longer hardwiring decision trees ‚Äî you‚Äôre exposing tools and letting the agent drive.

    Agents let you move logic from code into controlled language ‚Äî making systems more flexible.

    üß™ Microproject: Agentify Your App

    üß† Reflection Questions

    1. What logic can you move from backend code into a prompt?
    2. When should you not let the model pick tools?
    3. Where might agents introduce risk or fragility?

    ‚úÖ Best Practices for Agent Design

    ‚úÖ Tools must be deterministic, safe, and well-typed
    ‚úÖ Log every thought, action, observation
    ‚úÖ Use short tool names + descriptions for LLM clarity
    ‚úÖ Cap max steps ‚Äî agents can loop forever if you‚Äôre not careful
    ‚úÖ Fallback to static logic when confidence is low
    ‚úÖ Let tools fail gracefully ‚Äî model should adapt

    üîê Don‚Äôt give agents the kitchen sink. Give them power with guardrails.
